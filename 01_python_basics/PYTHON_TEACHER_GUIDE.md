## 08 Error Handling — `02_common_runtime_errors.py`

### 🎯 เป้าหมายของบท
ไม่ใช่ให้ "ท่องจำ error ทุกตัว" แต่คือฝึกให้นักเรียน **เห็น error แล้วบอกประเภทได้
→ เดาสาเหตุได้ทันที** ซึ่งเป็นทักษะ debug ที่ junior Data Engineer ต้องมีตั้งแต่วันแรก

### 📖 ลำดับการสอน

**ขั้นที่ 1 — สอน "อ่าน traceback" ก่อนเข้าเนื้อหา (สำคัญสุด)**
รัน error จริง **1 ตัวแบบไม่ใช้ `safe_run`** ให้เห็น traceback เต็ม ๆ แล้วสอนว่า:
- อ่าน **จากล่างขึ้นบน** — บรรทัดล่างสุดคือ `ErrorType: message` (สำคัญที่สุด)
- บรรทัดเหนือขึ้นไปบอก **เลขบรรทัด + ไฟล์** ที่พัง
- รู้แค่ "ประเภท" กับ "บรรทัด" ก็แก้ได้ 80% แล้ว

**ขั้นที่ 2 — เดินทีละ error แบบ "ทาย → รัน → เฉลย"**
อย่าอ่านโค้ดรวดเดียว ทำเป็นเกม: โชว์โค้ด → *(รอให้นักเรียนทายประเภท error)* → ค่อยรัน
helper `safe_run` ในไฟล์ออกแบบมาให้รัน error หลายตัวรวดเดียวโดยสคริปต์ไม่ตาย — เหมาะกับ demo สด

**ขั้นที่ 3 — จัดลำดับความสำคัญ (DE framing)**
ไม่ต้องให้น้ำหนักเท่ากันทุกตัว เน้น 2 ตัวแรกที่ junior DE เจอบ่อยสุด

### ⭐ ตารางจัดลำดับความสำคัญ

| Error | สถานการณ์จริงในงาน Data | จุดสอน / วิธีแก้ |
|-------|--------------------------|------------------|
| **TypeError** ⭐ | `"3" + 35000` — field จาก CSV เป็น **string หมด** | ต้อง cast ก่อนคำนวณ |
| **ValueError** ⭐ | `int("three")`, `int("")` — cell เสีย/ว่าง | cast แล้วต้องดัก error |
| KeyError | `product["price"]` — JSON ขาด field | ใช้ `.get("price")` |
| IndexError | `order_line[5]` — แถวมี column ไม่ครบ | เช็ก `len()` ก่อน |
| AttributeError | `"P001".to_upper()` — เรียกเมธอดผิดชื่อ | อ่านชื่อ method ให้ถูก (`.upper()`) |
| ZeroDivisionError | `revenue / 0` — เฉลี่ยจากจำนวนออเดอร์ = 0 | guard ก่อนหาร |
| NameError | พิมพ์ชื่อตัวแปร/column ผิด (`unit_prince`) | เช็กสะกด + ตัวพิมพ์ |
| FileNotFoundError | เปิด dataset ที่ไม่มีอยู่ | เช็ก path ก่อนเปิด |

> 🔑 ย้ำกับนักเรียน: **"raw CSV/JSON ทุกค่าเป็น string"** → `TypeError` กับ `ValueError`
> คือศัตรูอันดับ 1 ของ Data Engineer ใช้เวลากับ 2 ตัวนี้ให้มากที่สุด

### ✍️ Mini Exercise (ให้นักเรียนลงมือ)
โยนโค้ดพังให้ 4–5 ชิ้น ให้ **ทายประเภท error ก่อนรัน** แล้วค่อยแก้ เช่น:
```python
row = "P001,3,35000".split(",")
total = row[1] * row[2]        # ← error อะไร? (TypeError: can't multiply str)
# แก้:
qty = int(row[1])
price = int(row[2])
print(qty * price)             # 105000
```
ชุดโค้ดพังเพิ่มเติมให้ลอง: `int("N/A")` (ValueError), `{"sku":"P001"}["price"]` (KeyError),
`[1,2,3][5]` (IndexError), `None.strip()` (AttributeError)

### ⚠️ จุดที่ต้องเตือน / นักเรียนมักพลาด
- `# noqa: F821` ในตัวอย่าง NameError = แค่ปิด warning ของ linter (เพราะ **ตั้งใจ** พิมพ์ผิด) ไม่ต้องงง
- นักเรียนมักสับสน **TypeError** (ผิด *ชนิด*) กับ **ValueError** (ชนิดถูก แต่ *ค่า* ใช้ไม่ได้)
  → ตัวอย่างชัด: `int("three")` ชนิดถูก (เป็น str ที่ส่งให้ `int()` ได้) แต่ค่าแปลงไม่ได้ = ValueError
- บทนี้ **คู่กับ `01_try_except.py`** — สอนให้เห็น/รู้จัก error ก่อน แล้วค่อยสอนวิธี "ดัก" ด้วย try/except
- ปิดท้ายโยงไป **Mini Incident** ของ `00_bash_basics` ได้เลย — สถานการณ์เดียวกัน:
  อ่าน log เจอ error → ระบุประเภท → ไล่หา root cause

### คำถามนำสนทนา
- "ทำไม `"3" + 35000` ถึงพัง แต่ `"3" + "5"` ไม่พัง?" → โยงเรื่อง type ของ field จาก CSV
- "ถ้าไม่อยากให้โปรแกรมตายตอนเจอ cell ว่าง ควรทำยังไง?" → โยงไป try/except + `.get()`
